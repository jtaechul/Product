## 북 캐럿셀 자동화 시스템 (`book-carousel/`) — 기획 & 핵심 규칙

> **프로젝트 목표**: 인스타그램 책 리뷰 캐럿셀(카드뉴스 5장)을 자동 생성·검증·배포하는 시스템.
> 공포감·호기심 자극형 콘텐츠로 팔로워 확보, 쿠팡파트너스 어필리에이트로 수익화.
> 최종 배포 직전 텔레그램으로 사용자 확인 단계를 둔다.

### 전체 시스템 흐름

```
스케줄러(트리거) 또는 수동 실행
  → Claude API(책 선정 + 5페이지 캐럿셀 텍스트 + 캡션 생성)
  → 이미지 생성 API(페이지별 일러스트, Phase 2)
  → Claude 검증(0~100점 평가 → 기준 미달 시 재생성 루프)
  → 텔레그램으로 캐럿셀 내용 + 캡션 전송(사용자 확인)
  → 인스타그램 자동 포스팅(Phase 4)
  → 팔로워가 게시물 보고 DM으로 키워드 전송
  → Instagram DM 자동 감지 → 책 제목 + 쿠팡파트너스 링크 자동 회신(Phase 5)
```

### 캐럿셀 5페이지 구조

- **1페이지(훅)**: 공포감·위기감·호기심 자극. `"당신이 이 내용을 알지 못한다면..."` 형태
- **2페이지(문제)**: 문제 상황 제시 (현실감 있는 수치·사례)
- **3페이지(심각성)**: 문제의 심각성 강조 (역사적 배경·통계, "대부분은 이 사실을 모른다")
- **4페이지(실마리)**: 책의 해결 방향 암시 (완전한 답은 남겨 호기심 유지)
- **5페이지(CTA)**: 책 제목 미노출. `"DM으로 '키워드'를 보내주세요"` 형태의 DM 유도

### 캡션 + 해시태그 규칙

- 첫 줄: 호기심 자극 단문/질문 (책 제목 직접 노출 금지)
- 2~3줄: 캐럿셀 핵심 내용 요약 (간결, 반복 금지)
- 마지막 줄: DM 유도 (`DM으로 '키워드'를 보내주세요`)
- 해시태그: 정확히 3개
- 노골적 판매 표현 금지, 우회적 표현 사용

### 어필리에이트 링크 처리 (Phase 5)

- 캐럿셀에 책 제목·링크 직접 노출 금지 (DM 회신으로만 제공)
- Claude가 쿠팡파트너스에서 해당 책 + 관련 상품 링크 자동 검색·추출
- 관련 상품 1~2개로 제한 (건강책→영양제, 수면책→수면용품 등)

### DM 키워드 매핑 예시

| DM 키워드 | 연결 카테고리 |
|---|---|
| 경제 | 투자·부동산·비트코인 |
| 심리 | 심리·관계·자존감 |
| 건강 | 건강·영양·수면 |

### 운영 원칙

1. **AI 자동 추천 모드**: 카테고리/이슈 선택 → Claude 책 3~5권 제안 → 사용자 선택 → 캐럿셀+캡션 자동 생성
2. **수동 입력 모드**: 카테고리 + 책 제목 (+ 선택: 저자, 핵심메시지) → Claude 자동 분석 → 캐럿셀+캡션 생성
3. **검증 루프**: Claude 자동 평가(0~100점) → 기준 미달 시 재생성 → 기준 초과 시 텔레그램 발송
4. **콘텐츠 전략**: 초기 3개월 베스트셀러 위주, 이후 사회이슈 연계 도서로 "인사이트 큐레이터" 포지셔닝

### API 엔드포인트

| 경로 | 기능 | 모델 |
|---|---|---|
| `POST /api/suggest` | 카테고리/이슈 → 책 3~5권 제안 | Claude opus |
| `POST /api/analyze` | 책 제목+저자 → 핵심메시지·독자층 분석 | Claude sonnet |
| `POST /api/generate` | 전체 정보 → 5페이지 캐럿셀 생성 | Claude opus |
| `POST /api/generate-caption` | 캐럿셀+책정보+DM키워드 → 캡션+해시태그 생성 | Claude sonnet |
| `POST /api/validate` | 생성된 캐럿셀 품질 검증 (0~100점) | Claude sonnet |
| `POST /api/regenerate` | 피드백 반영 재생성 | Claude opus |
| `POST /api/send-telegram` | 캐럿셀+캡션 텔레그램 발송 | — |

### 구현 단계

- **Phase 1** (완료): 책 선정(AI추천/수동) + 5페이지 텍스트 생성 + 캡션/해시태그 생성 + 검증 + 텔레그램 발송
- **Phase 2** (완료): 이미지 생성 API 연동 (Pollinations.ai, 무료) + Claude 이미지 프롬프트 생성 + 카드별 일러스트 자동 적용 (`/api/generate-images`)
- **Phase 3** (예정): 검증 루프 자동화 + 재생성 자동 트리거
- **Phase 4** (예정): 인스타그램 자동 포스팅 (Upload-Post MCP)
- **Phase 5** (예정): Instagram DM 자동 회신 + 쿠팡파트너스 링크 추출 + 구글 클라우드 스케줄러

### 배포 규칙

- Cloudflare Workers (`book-carousel.jtaechul.workers.dev`) 자동 배포
- 브랜치: `claude/book-carousel-auto-upload-xpwihz` push 시 자동 배포
- 필요 시크릿: `ANTHROPIC_API_KEY`, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`

> ## ⭐ 핵심 응답 원칙 (모든 작업에 항상 적용)
> **매 작업이 끝나면, 마지막에 "방금 한 일"을 비전문가도 이해할 수 있게 아주 간단히
> 정리한다.** 규칙:
> - 3~5줄 이내, 쉬운 한국어. 전문용어는 풀어서 또는 괄호로 설명.
> - "무엇을 / 왜 / 결과" 위주. 코드·수치 나열 최소화.
> - 길고 자세한 내용이 필요하면 그건 정리 *뒤*에 따로 둔다(요약이 먼저).
>
> ## ⭐ 터미널/SSH 안내 원칙 (사용자는 터미널 초보)
> 사용자는 SSH·터미널·명령어를 다룰 줄 모른다. 서버(클라우드) 작업을 안내할 때는:
> - **키 하나, 한 단계까지** 빠짐없이. (예: "`q` 를 누르세요", "Ctrl 키를 누른 채 X")
> - 화면에 뜬 낯선 것(`(END)`, 편집기, 에러 등)은 **무슨 화면인지 + 어떻게 빠져나오는지** 설명.
> - **왜 하는지 원리**를 한두 줄로 쉽게 곁들인다(비유 환영).
> - 막히면 "그 화면을 캡처해서 보내달라"고 안내하고, 추측 대신 확인 후 진행.

> ## ⭐ 링크 전달 원칙 (필수)
> 사용자에게 URL을 전달할 때는 반드시 마크다운 클릭 가능 링크 형식으로 작성한다.
> - 형식: `[링크 설명](https://전체URL)`
> - 예시: `[캐럿셀 생성기](https://book-carousel.jtaechul.workers.dev)`
> - 절대 URL을 그냥 텍스트로만 쓰지 않는다 — 클릭이 안 되기 때문.

> ## ⭐ 아이콘/이모지 금지 원칙 (필수)
> **OS 기본 이모지(이모티콘)를 절대 쓰지 않는다.** 버튼·HUD·라벨·활동/매체/아이템/엔딩 등 게임 화면
> 어디에도 🎬📂🚪❤🧠💰⭐ 같은 시스템 이모지를 넣지 않는다(기기마다 모양이 다르고 싸구려처럼 보임).
> - 아이콘이 필요하면 **동일 화풍의 커스텀 일러스트(PNG)** 또는 **Pixi 벡터(Graphics)로 직접 그린 도형**만 사용.
> - 텍스트만으로 충분하면 아이콘 없이 깔끔한 글자만 둔다.
> - 기존 코드에 남아 있는 이모지는 발견하는 대로 커스텀 아이콘/벡터/텍스트로 교체한다.

## 자동 배포 규칙 (기본 동작)

**코드 수정 작업이 완료되면 항상 자동으로 다음을 수행한다 (사용자가 명시적으로 막지 않는 한):**

1. 작업 브랜치(예: `claude/*`)에 커밋 후 push
2. `main` 브랜치로 checkout → 최신 pull
3. 작업 브랜치를 `main`에 `--no-ff` 머지
4. `main` push → GitHub Pages 자동 배포 트리거
5. 사용자에게 다음을 보고:
   - 머지된 커밋 목록
   - 배포 URL: `https://jtaechul.github.io/Product/spotlight/` (SPOTLIGHT 기준)
   - 브라우저 캐시 강력 새로고침 안내 (Ctrl/Cmd+Shift+R)
6. 작업 브랜치로 다시 checkout (다음 작업 준비)

**예외:**
- 사용자가 "머지 금지", "PR만 생성해", "main 푸시 금지" 등을 명시한 경우
- 머지 충돌 발생 시: 자동 해결 시도 후 실패하면 사용자에게 보고
- 작업이 중단됐거나 QA 미완료 상태인 경우

**개발 규칙 (매 단계 공통):**
- **매 단계 완료 후 반드시 테스트**: 코딩 완료 → 로컬 서버 기동(`python3 -m http.server 8000`) →
  브라우저(특히 모바일 세로)에서 정상 작동 검증 → 문제 발견 시 즉시 수정 → 검증 통과 후 커밋/배포
- **데이터 분리**: 활동·작품·이벤트·엔딩·애니메이션 수치는 `js/data/*.js`에 모아 플레이 테스트로 미세 조정

## 2. 스토리 & 배경

평범한 고등학교 1학년 **주인공**은 길거리 캐스팅 명함을 받는다. 작은 소속사의 신입 매니저 **‘한지원’** 이 건넨 명함엔 이렇게 쓰여 있다 — *“네 안에 빛나는 무언가가 보여.”*

주인공은 그날부터 배우를 꿈꾸지만, 동시에 공부도 친구도 가족도 건강도 챙겨야 하는 평범한 학생이다. 매니저는 곁에서 조언하고 출연 기회를 물어다 주지만, **모든 선택은 주인공의 몫**이다.

3년의 시간을 어디에 쓸 것인가. 그 누적이 졸업하는 날의 실력과 인격을 만들고, 그 토대 위에서 **40년에 걸친 배우 인생**이 펼쳐진다.

### 아트 디렉션

- **밝고 생기 있는 청춘 애니메이션 톤** (우마무스메처럼 채도 높은 캐릭터 + 부드러운 배경)
- 캐릭터가 화면의 **주인공**: 세로 화면 중앙 상단을 캐릭터가 크게 차지하고, 늘 살아 움직임(idle·표정·활동 모션)
- 컬러: 파스텔 베이스 + 코랄/민트 포인트, 골드(스타성·수상) 강조색
- “지금 내 캐릭터가 무언가 열심히 하고 있다”는 느낌을 항상 유지 → 육성 게임의 애착 형성

## 4. 기술 스택 & UI 완성도 전략

> ⚠️ **핵심 결정 (v3.1)**: 게임 화면을 DOM+CSS로 그리지 않는다. 활동마다 짧은 애니메이션이 계속 들어가는
> 우마무스메식 게임은 DOM에선 끊김·발열·최적화 한계가 생긴다. **화면은 GPU(WebGL)로 그리는 PixiJS로
> 렌더링하고, 이를 Capacitor로 진짜 설치형 앱(iOS/Android)으로 포장**한다. "웹앱이라 구린 것"이 아니라
> "DOM으로 게임을 그려서 구린 것"이므로, 렌더링 방식을 바꿔 그 문제를 근본적으로 없앤다.

### 기술 스택

| 영역 | 채택 | 비고 |
|---|---|---|
| **게임 화면 렌더링** | **PixiJS (WebGL/WebGPU 2D 렌더러)** | DOM이 아닌 GPU 렌더링. 스프라이트·애니 수백 개도 60fps. UI·HUD도 Pixi 위에 구성 |
| **캐릭터 애니메이션** | **Live2D Cubism** (메인) / **Rive** (전환·이펙트) | "늘 살아 움직이는 캐릭터 + 활동별 모션" = 우마무스메식. Live2D는 일러스트 1장을 관절처럼 구동, Rive는 "활동 선택→짧은 애니" 상태 전환에 가볍고 적합 |
| **언어/모듈** | **JavaScript (ES 모듈) · 빌드 없음** | 브라우저 native ES module. PixiJS는 `vendor/pixi.min.mjs` 로컬 파일을 importmap으로 로드 → **빌드/번들 단계 없이 GitHub Pages 날것 서빙으로 바로 동작** (Vite/TS 빌드는 Pages 미변환 → 폐기) |
| **아트(중요)** | **고퀄 래스터 일러스트(PNG)** — 캐릭터·배경·UI 패널·활동 썸네일 | ⚠️ **SVG 손그림 금지**: 우마무스메급 퀄리티는 그려진 일러스트(PNG)로만 가능. AI(Canva·Adobe·Higgsfield)로 동일 화풍 생성 → `assets/`에 PNG로 저장 → PixiJS Sprite로 조립. (SVG는 단순 아이콘 정도만) |
| **캐릭터 애니메이션** | **Live2D Cubism** / **Rive** (후반 적용) | 일러스트를 관절처럼 구동. 초기엔 정적 PNG → 점진 적용 |
| **데이터** | 활동·작품·이벤트·엔딩 수치를 **JS 데이터 파일**로 분리 | `src/data/*` (수치 분리, 플레이 테스트로 조정) |
| **사운드** | **Howler.js**(CDN/vendor) 또는 Web Audio API | BGM·효과음 관리 |
| **저장** | `localStorage` (앱화 시 Capacitor Preferences) | 세이브·로드·엔딩 도감 |
| **앱 패키징(후반)** | **Capacitor** | 완성된 정적 사이트(`spotlight/`)를 그대로 감싸 iOS/Android 앱화 |
| **배포** | **GitHub Pages** (`/Product/spotlight/`) — main/root 날것 서빙 | 빌드 산출물 불필요, 소스가 곧 배포본 |

> 💡 **왜 PixiJS인가**: ① 전부 코드라 AI(Claude)가 처음부터 끝까지 제작 가능(게임 엔진 에디터처럼 사람이
> GUI를 직접 만질 필요 없음) ② GPU 렌더링이라 DOM 웹앱의 끊김·발열이 사라짐 ③ Capacitor로 진짜 앱이 됨.
> → 비전문가 + AI 협업 + 애니 다발 게임의 스위트스폿.

### 렌더링/씬 구조 원칙

- 화면(타이틀·캐릭터생성·메인스케줄·평가·엔딩 등)은 **Pixi 씬(Scene) 단위**로 전환
- 캐릭터는 **상시 idle 모델 1개를 메인 스테이지에 고정**, 활동 선택 시 해당 모션을 트리거(섹션 14)
- HUD(체력·멘탈·돈·팬)·활동 카드·버튼도 Pixi 컨테이너로 그리되, **터치 타깃 48px↑·하단 엄지존** 규칙(섹션 16) 준수
- 텍스트가 많은 패널(필모·설정 등)은 Pixi 텍스트 또는 가벼운 DOM 오버레이를 선택적으로 혼용 가능(성능 우선 판단)

### ⭐ UI 완성도 확보 전략

> NIGHT HUNTER가 투박했던 원인은 3D 기본 도형 모델링. 2D 게임의 완성도는 (1) 디자인 시스템 (2) 일러스트·애니메이션 품질 (3) GPU 렌더링으로 결정되며, 모두 통제 가능하다.

1. **디자인 시스템 선고정** — 색·폰트 스케일·여백·카드 컴포넌트를 코딩 전에 토큰화(디자인 시트 팔레트 기준)
1. **AI 일러스트 일관 생성** — 주인공 표정·활동 포즈·배경·엔딩을 **Canva · Adobe(Firefly) · Higgsfield** 로 동일 화풍 시리즈 생성 → Live2D/Spine 리깅 소스로 사용
1. **메인(세로) 화면 Figma 목업 우선** — 룩 확정 후 PixiJS로 구현
1. **연출은 Live2D/Rive로 GPU 처리** — 활동별 짧은 애니를 끊김 없이, 무겁지 않게 생동감 확보

## 5. 핵심 게임 메커닉

```
📅 한 달(1턴) 진행
├ 1. 매니저 인사 + 이번 달 들어온 출연 제안
├ 2. 활동 2개 선택 (연기/학업/인성/매력/회복 또는 '작품 출연')
│     → 선택 시 메인 캐릭터는 idle 유지(슬롯에만 담김). 활동 이미지는 아직 안 나옴
├ 3. '다음 달 진행' 버튼 → 선택한 활동 이미지(포즈)+관련 대사 연출 재생
│     (가끔 계절 특별 대사가 앞에 삽입됨)
├ 4. 결과 정산 (스탯 ↑ / 돈·체력·멘탈 변동 / 인연 ↑)
├ 5. 출연했다면 → 시청자 평가 연출(호평/혹평 → 스탯·팬 반영)
├ 6. 랜덤 이벤트 (선택지 → 인성·플래그)
└ 7. 다음 달 (1턴 경과, 마일스톤 체크)

   36턴(3년) → 졸업 → 스탯·필모·인성·인연 종합 → 40년 커리어 엔딩
```

## 8. 난이도 밸런스 & 성장 곡선

게임의 긴장감과 적정 난이도를 위해 **활동 예산 → 성장 곡선 → 난이도 모드**를 수치로 설계한다.

### 활동 예산 (36턴 기준)

- 총 슬롯 = 36턴 × 2 = **72슬롯**
- 회복(휴식·가족·우정) 권장 ≈ 14슬롯 / 작품 출연 ≈ 8~10슬롯 → **순수 훈련 ≈ 48~50슬롯**
- 즉, “모든 능력치를 다 키울 수는 없다”는 자원 압박이 설계의 핵심

### 성장 소프트캡 (체감 곡선)

스탯이 높을수록 상승 효율이 줄어 한 능력치를 100까지 올리기 어렵게 만든다 → 분산을 유도(우마무스메식).

|현재 스탯 구간|상승 배율|
|--------|-----|
|0 ~ 39  |×1.0 |
|40 ~ 59 |×0.7 |
|60 ~ 79 |×0.5 |
|80 ~ 100|×0.3 |


> 예) 연기력을 80까지 올리려면 약 22슬롯, 100까지는 약 35슬롯 소모. → 한 스탯에 올인하면 다른 스탯이 비게 되어, **연기+인성+학업을 모두 요구하는 국민배우 루트가 가장 어렵게** 설계된다.

### 빌드별 도달 난이도

|플레이 스타일        |결과 경향      |난이도    |
|---------------|-----------|-------|
|연기 올인          |연기파 배우·신스틸러|★★☆    |
|외모·인지도 집중      |스타성 배우     |★☆☆    |
|균형(연기+인성+학업+매력)|**국민 대배우** |★★★    |
|무계획·회복 부족      |슬럼프·무명     |(자연 실패)|

### 난이도 모드 — *현재 미사용(제거됨)*

> 변경(결정): 난이도 모드(쉬움/보통/어려움)는 **삭제**했다. 시작 돈은 100,000 고정, 스탯 상승 배율 ×1.0.
> 마일스톤(고1·고2 말 인지도 목표)은 **방출 없이 달성/미달 안내 토스트만** 띄운다.

|모드      |시작 돈   |스탯 상승|출연 기대치|실패 확률|마일스톤           |
|--------|-------|-----|------|-----|---------------|
|쉬움      |150,000|×1.2 |완화    |낮음   |없음             |
|보통(기본)  |100,000|×1.0 |표준    |표준   |경고만            |
|어려움(우마식)|70,000 |×0.9 |상향    |높음   |**필수(미달 시 방출)**|

### 자원 곡선 가이드 (밸런싱 기준)

- **돈**: 용돈 36×5만 = 180만 < 매달 레슨비(6~8만)×36 → 출연료·알바로 메워야 함(적자 압박)
- **체력**: 활동 1회 평균 -10, 턴당 약 -20 → **3~4턴마다 휴식 1슬롯** 필요하도록 회복량 조정
- **멘탈**: 혹평·고강도 누적 시 하락 → 후반 슬럼프 방지하려면 회복 활동 필수
- 모든 수치는 `data/*.js`에서 한곳에 모아 **플레이 테스트로 미세 조정** (개발 9단계)

## 10. 자원 관리

```
활동·출연 → 체력·멘탈 소모 → 휴식/가족/친구로 회복 → 다시 도전
            ↑ 과욕 → 슬럼프·부상으로 한 달 손실
            ↑ 혹평 → 멘탈 큰 타격 → 회복 필요
```

- **돈**: 용돈 + 출연료 + 알바 vs 레슨비 → 늘 빠듯하게(선택과 집중 강제)
- **체력/멘탈**: 회복 슬롯을 아예 안 넣으면 후반 붕괴하도록 밸런싱
- 매 턴 “조금 무리할까 / 안전하게 갈까” 판단이 핵심 재미

## 11. 작품 출연 & 시청자 평가 시스템

게임의 핵심 “레이스”(우마무스메 경주에 해당). 매월 변동하는 출연 제안에 출연하면 **스탯에 따라 시청자 반응이 갈린다.**

### 작동 방식

1. 매달 매니저가 **1~3개 출연 제안**을 가져옴(매체·배역 매번 다름, 인맥·인연·인지도로 질 상승)
1. 출연 시 활동 슬롯 1개 사용 → **훈련 대신 실전**을 택하는 트레이드오프
1. `종합 연기점수 = (연기력·감정표현·발성 + 배역 가중 스탯) vs 매체 기대치`
1. 평가 등급 산출 → 보상/페널티 + **필모그래피 기록**(엔딩 재료)

### 시청자 평가 등급

|등급         |기대치 대비|결과                             |
|-----------|------|-------------------------------|
|🌟 인생 연기(극찬)|크게 초과 |팬·연기력 大↑, **수상 후보 플래그**, 멘탈 +15|
|👍 호평       |충족~초과 |팬·연기 경험치↑, 멘탈 +8               |
|😐 무난       |약간 미달 |소폭 팬↑                          |
|👎 혹평       |크게 미달 |멘탈 -12, 차기 제안 감소               |

### 매체 라인업 (매월 변동 · 학년 따라 확장)

|매체          |등장  |요구(대략)              |출연료 |호평 시 핵심 보상           |

> 📌 **스탯 → 반응 연동**: 같은 작품도 연기력·감정표현이 높으면 극찬, 낮으면 혹평. 배역별 가중 스탯(정극=감정표현, 비주얼=외모, 무대=발성·가창)을 둔다. 인성이 받쳐줘야 호평이 “호감”으로 누적되어 팬이 안정적으로 증가(인성 낮으면 인기는 떠도 평판 플래그 악화).

### 평가 연출

- 출연 → “방영/개봉” 컷 → **시청자 댓글이 주르륵 올라오는 연출**(등급별 톤 변화) → 등급 등장 → 스탯 변동 → 필모 한 줄 추가

## 13. 랜덤 이벤트 & 선택지 시스템

매 턴 일정 확률 발생. 선택지가 스탯·자원·플래그를 바꾸며 특히 **인성**에 작용.

**긍정**: 선배 멘토링(연기+8, 인성+3) / 팬레터(멘탈+15) / 후배 돕기(인성+8, 인맥+3)
**부정**: 슬럼프(상승 -50%) / 악플(멘탈-15, 대응 분기) / 부상(체력-30) / 촬영장 갈등(인성 분기)

```
[이벤트] 동료가 NG를 연발해 촬영이 길어진다.
 ┌ A. 다독인다     → 인성 +6, 인맥 +3
 ├ B. 한숨 쉰다    → 인성 -5, 평판 플래그
 └ C. 내 연습 집중 → 연기력 +3
```

- 숨은 플래그(존경받는 배우 / 논란의 배우)가 엔딩 서술에 직접 반영
- 논란 플래그 누적 시 인기가 높아도 “구설수 배우” 계열로

## 15. 엔딩 시스템

**모든 엔딩은 배우의 인생.** 졸업 시점의 스탯·필모·인성·인연·플래그를 종합해 **데뷔 후 40년 커리어를 60세 시점에서 회고하는 서술형 엔딩**으로 보여준다. 화면에는 **수치를 노출하지 않고** 일러스트 + 서사만 제시한다.

### 모듈형 엔딩 생성 (다양도 극대화)

고정 결말 대신 **3개 축의 조합**으로 회차마다 다른 엔딩 서사를 생성한다.

```
[분야]  영화 / 드라마 / OTT / 뮤지컬 / 연극무대 / 예능·버라이어티 / 광고
[급]    국민배우 / 톱 주연 / 주연급 / 명품 조연 / 신스틸러 / 단역·무명
[특성]  연기파 · 스타성 · 국제파(한류) · 인격자 · 대기만성 · 논란 · 후학양성
   └→ 분야는 '호평받은 필모 분포', 급은 '연기·팬 종합', 특성은 '인성/플래그/인연'으로 결정
   └→ 실제 호평작 제목·수상 이력을 본문에 끼워 넣어 매 회차 다른 회고로 출력
```

### 대표 엔딩 (15종+)

> 외모(looks) 반영: 화면에 비치는 **대중 스타 엔딩**(국민 대배우·안방극장의 별·글로벌 한류·스타성)은
> 외모 임계값을 요구한다. 외모가 낮으면 이쪽 대신 **연기파(영화계 거장)·성격파·연극·카메라 뒤(감독/작가/PD/제작자)**
> 로 갈린다 → "외모가 약하면 화려한 스타는 어렵지만, 실력파·창작자의 길이 열린다". (`data/endings.js`)

|엔딩             |결정 경향(내부 기준, 비노출)                 |
|---------------|----------------------------------|
|🏆 국민 대배우       |연기 高 + 인성 高 + 학업 中↑ + 영화·드라마 양쪽 수상|
|🎬 영화계의 거장      |연기력·감정표현 高 + 영화 필모·영화상 위주         |
|📺 안방극장의 별      |발성·외모 + 드라마/OTT 필모·연기대상           |
|🍿 글로벌 한류 스타    |OTT·해외 인지도 플래그 + 팬 高              |
|🎵 뮤지컬 디바/프린스   |가창·발성 高 + 무대 평론가 플래그              |
|🎭 연극 무대의 거목    |연기력·발성 高 + 무대 위주, 대중 인지도는 낮음      |
|🌟 스타성 배우       |외모·팬 高 + 연기력 中 + 인성/학업 비어 있음      |
|🎪 예능 대세        |댄스·가창·인맥 + 버라이어티 활약               |
|🧩 천의 얼굴 성격파    |감정표현·인성 高 + 다양한 장르 필모             |
|🎭 명품 조연·신스틸러   |연기력 中上 + 다양한 필모, 주연급 인기는 아님       |
|🌱 대기만성 늦깎이     |초중반 부진했으나 후반 급성장                  |
|🎬 감독 겸 배우      |학업·인성 高 + 연기력 高 (특수)              |
|🎓 후학을 키운 대배우   |국민배우 조건 + 인성 최상 + 연기선생 인연 만점(특수)  |
|🌙 묵묵한 무명·조연    |전반 낮으나 끝까지 포기 안 함(따뜻한 엔딩)         |
|💔 구설수 속에 사라진 배우|논란 플래그 + 인성 低                     |

### 엔딩 화면 구성

1. 페이드 → **엔딩 일러스트**(노년 배우·시상식·필름 분위기) + 타이틀
1. **서술형 40년 회고** (수치 없이): 데뷔작 → 전성기 → 대표작 → 수상 → 60세 현재
1. **대표 필모그래피** (플레이 중 호평작이 실제 나열)
1. **수상 이력** (플래그로 생성)
1. **함께한 사람들** (인연 80↑ 인물 등장)

### 모바일 최적화 규칙

- **세로 고정 레이아웃**, 주요 액션 버튼은 **하단 50% 엄지존**에 배치
- **세로 방향 강제 고정(안내문 없음)**: 지원 기기(Android/PWA)는 `screen.orientation.lock('portrait')` 시도,
  잠금 불가 환경(iOS Safari 등)은 **가로로 들면 Pixi 스테이지를 90° 회전**해 세로 화면을 그대로 유지한다
  (DOM 캔버스 회전이 아니라 스테이지 회전 → Pixi 히트테스트가 회전을 반영해 터치 좌표가 정확). `SceneManager.resize()`.
- 탭 타깃 최소 48×48px, 버튼 간 간격 넉넉히
- **hover 의존 금지** — 모든 정보는 탭/롱프레스로 접근
- 활동 선택은 **카드 탭 → 슬롯에 담기 → 확정** 방식(오조작 방지)
- 패널 전환은 풀스크린 시트 + 스와이프 닫기
- 폰트: 기본 16px↑, `rem` 기반 반응형 / 안전영역(`env(safe-area-inset)`) 대응
- PC: 세로 레이아웃을 중앙 정렬(모바일 프레임) 또는 좌우 여백 활용한 확장

> `spotlight/assets/portraits/heroine_design_sheet_red.png`(붉은 머리). 모든 일러스트·스프라이트는

- **공통 컨셉**: 밝고 청순한 청춘 애니메이션 톤, 큰 눈·또렷한 하이라이트, 부드러운 채색
- **의상(기본 교복)**: 네이비 블레이저 + 흰 셔츠 + **파란 리본 타이** + 하늘색 체크 플리츠 스커트
  + 네이비 무릎 위 양말 + 브라운 로퍼 (교복은 SPOTLIGHT의 시그니처 룩)
- **머리/눈 컬러 변형 2종 (캐릭터 생성 시 선택 가능하게)**:
  - **버전 A — 브라운**: 갈색 웨이브 롱헤어 + **파란 꽃/리본 머리장식**, 갈색 눈 → 따뜻·청순 인상

- **필수 포즈 세트(디자인 시트 기준 5종)**: ① 정면 ② 후면 ③ 측면 ④ ¾ 시점 ⑤ 동적 포즈
- **표정 5종**: 기쁨 / 슬픔 / 놀람 / 뿌듯 / 풀죽음 (레이어 교체 + 페이드)
- **활동 포즈 9종**: 9번 활동 표(연기·보컬·댄스·헬스·독서실·봉사·가족·휴식·출연)에 매핑

### 기타 캐릭터 & 배경

- **매니저(한지원)·라이벌(유세아)·연기선생(노교수)·친구**: 각 표정 3~4종
- **엔딩 일러스트**: 15종(노년 배우 분위기) — Canva/Higgsfield/Adobe 동일 화풍 생성
- **배경**: 학교 / 학원 / 촬영장 / 무대 / 시상식 / 집
- 프로토타입은 단색 실루엣·이모지 → 후반에 일러스트·애니메이션 교체

### ⭐ 프레임 안 얼굴 배치 규칙 (필수·기본 원칙)

- **얼굴은 위·왼쪽·오른쪽을 절대 크롭하지 않는다. 오직 아래(목젖)만 (사각 프레임은 직선으로) 크롭한다.**
  - 위: 머리끝(정수리)이 잘리지 않게 보존.
  - 좌·우: 얼굴/머리 양옆이 잘리지 않게 보존 → 자연히 좌우에 약간의 여백이 생긴다.
  - 아래: 목젖 근처에서 직선으로 잘라 프레임 하단을 채운다.
- **좌측·우측 여백 폭은 항상 거의 같아야 한다(= 가로 중앙 정렬).** 인물 얼굴 중심선을 프레임 가로 중심에 맞춘다.
- 구현: 머리끝~목젖 세로 구간을 프레임 높이에 맞춰 채우면(height-fit) 얼굴이 세로로 길어 좌우는
  자동으로 여백이 남고 위·좌·우가 잘리지 않는다(`_faceInRect`). 인물별 머리끝·목젖·가로중심은
  `data/bonds.js`의 `lm`(top·throat·cx 비율)에 저장.
